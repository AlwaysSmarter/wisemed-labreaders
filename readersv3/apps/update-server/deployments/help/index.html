WiseMED App Update Server

Ghid complet pentru publicarea aplicațiilor, generarea versiunilor, distribuția linkurilor de download și configurarea serverului de update.

1. Ce face aceasta aplicatie

App Update Server este consola locala prin care administrezi pachetele de update pentru aplicatiile WiseMED Reader. Aplicatia ofera:

  • catalog de aplicatii publicate
  • istoric de versiuni pe OS si arhitectura
  • generare automata de release din sursele locale
  • publicare pachete manuale sau generate
  • linkuri publice de download cu token unic
  • endpoint public de check-update pentru readere

2. Meniul principal

Aplicatii

Defineste aplicatiile pentru care serverul publica update-uri. Fiecare aplicatie are un App ID, nume afisat si descriere.

Versiuni

Administreaza versiunile publicate, filtreaza pe canal, sistem de operare si arhitectura, creeaza release-uri noi si genereaza linkuri publice.

Setari

Configureaza prefixul public de download, directoarele de lucru si credentialele WiseMED API folosite pentru autentificare si verificare.

Ajutor

Deschide acest ghid direct din meniul principal atunci cand ai nevoie de documentatie operationala sau de troubleshooting.

3. Flux recomandat de lucru

  1. Deschide Aplicatii si verifica daca aplicatia exista deja in catalog.
  2. Selecteaza aplicatia si mergi in Versiuni.
  3. Alege Canal, OS tinta si Arhitectura.
  4. Apasa Make release daca vrei sa generezi automat pachetul din sursele locale.
  5. Alternativ, foloseste Salveaza versiunea si incarca manual un pachet in campul Installer / pachet upload.
  6. Dupa creare, selecteaza versiunea si apasa Genereaza one-time-token pentru un link public de download.

Fluxul automat foloseste versiunea embedded in binar si genereaza pachetul de update fara a include config.yaml, baze de date sau fisiere runtime.

6. Make release

Make release foloseste sursele locale ale proiectului readersv3 pentru a genera automat un pachet publicabil.

  • foloseste binarul compilat cu versiune embedded
  • genereaza payload curat pentru update
  • pentru Windows genereaza automat si installerul NSIS .exe
  • include binarul si deployments/config.install.yaml
  • exclude config.yaml, baze de date, loguri, cache si fisiere runtime

Installerul Windows generat automat ramane separat de pachetul de update. Reader-ele folosesc in continuare arhiva de update, iar operatorul poate descarca installerul din formularul versiunii create.

Pentru ca fluxul sa functioneze, serverul trebuie sa ruleze pe o masina care are acces la repo-ul local si la toolchain-ul Go necesar pentru build.

7. Linkuri publice de download

Linkurile de download sunt generate sub forma:

https://prefix-exemplu.ro/api/public/download/11?token=...

Partea configurabila este tot ce se afla inainte de /api/public/download/....

Setarea corecta

  • Daca serverul este accesibil local, poti folosi de exemplu http://127.0.0.1:19090.
  • Daca serverul este publicat in retea sau prin reverse proxy, seteaza URL-ul extern, de exemplu https://updates.exemplu.ro.

Campul din setari care controleaza acest lucru este Prefix public link download. Daca este completat, are prioritate fata de protocol, host si port.

10. Troubleshooting

Nu se pot genera linkuri corecte

  • Completeaza Prefix public link download cu URL-ul exact prin care clientii ajung la server.
  • Daca folosesti reverse proxy, nu lasa doar 127.0.0.1, pentru ca linkul va fi inutil in afara masinii locale.

Make release esueaza

  • Verifica existenta repo-ului local readersv3 pe masina serverului.
  • Verifica toolchain-ul Go si permisiunile pe directoarele temporare si pe files_dir.

Readerul nu vede update-ul

  • Verifica daca app_id din reader corespunde cu App ID din update-server.
  • Verifica daca versiunea publicata corespunde cu OS-ul si arhitectura readerului.
  • Verifica daca versiunea embedded in binarul readerului este mai mica decat cea publicata.

Download-ul esueaza

  • Verifica daca fisierul exista in files_dir.
  • Verifica daca prefixul public de download este accesibil din reteaua clientului.
  • Verifica logurile pentru erori la token, acces sau checksum.

11. Bune practici

  • Pastreaza App ID stabil pe toata durata de viata a produsului.
  • Publica separat pe fiecare combinatie de OS si arhitectura necesara.
  • Foloseste Mandatory doar pentru update-uri care chiar trebuie aplicate imediat.
  • Seteaza un URL public real in Prefix public link download daca linkul ajunge la clienti externi.
  • Nu publica in pachete fisiere runtime sau configuratii active ale clientului.
